# OfferFlow / Offer来了 · 多 AI 协作职责文档

## 1. 目标

本项目采用多 AI 协作，但必须避免多头指挥和范围膨胀。

核心原则：

> 一个产品方向，一套任务卡，一步一验收。

任何 AI 都不能绕过任务卡自由发挥。

---

## 2. 一句话理念

把 AI 当成会遗忘、会漂移、会越界的协作者来治理，而不是当成聪明的补全。

所有文档和工具的存在意义只有三个：

1. 约束行为
2. 沉淀决策
3. 保证跨会话能接力

凡是不服务这三件事的 AI 文档，都是摆设。

---

## 3. 分域唯一信源

OfferFlow 不使用单个大 SPEC，而采用“分域唯一信源”。

| 领域 | 唯一信源 |
|---|---|
| 产品定位 / 目标用户 / 产品边界 | docs/v0.1/product.md |
| 开发步骤 / Step 顺序 | docs/v0.1/dev-plan.md |
| 当前任务卡 / P0 范围 | docs/v0.1/task-cards.md |
| 验收标准 | docs/v0.1/acceptance.md |
| AI 协作分工 / 工作流规则 | docs/v0.1/roles-and-workflow.md |
| 当前进度 / 当前允许做什么 | docs/v0.1/progress.md |
| 重要决策 / 决策来源 / 拍板记录 | docs/v0.1/decision-log.md |

如果文档之间存在冲突：

1. 先看用户最新明确指令
2. 再看 docs/v0.1/progress.md 判断当前阶段
3. 再看 docs/v0.1/decision-log.md 判断已拍板决策
4. 再看 docs/v0.1/task-cards.md 判断当前任务范围
5. 再看 docs/v0.1/acceptance.md 判断完成标准
6. 不允许静默选择其中一份继续开发

## 4. 角色分工

### 4.1 GPT

角色：

产品经理 / 需求负责人 / 边界裁判。

职责：

1. 定义产品方向
2. 收束 v0.1 范围
3. 判断功能是否进入 P0
4. 拦截范围膨胀
5. 审查 AI 输出是否偏离需求
6. 帮助用户做最终决策

GPT 不直接负责仓库代码实现。

### 4.2 Claude.ai

角色：

首席架构师 / 技术顾问 / 文档顾问。

职责：

1. 提供架构建议
2. 提供落地整理文档
3. 提供开发前任务拆解
4. 辅助判断技术风险
5. 可产出候选代码，但候选代码必须进入本地仓库后由 CC / Codex 验证

Claude.ai 不应继续绕过仓库直接推进主线开发。

### 4.3 Claude Code / CC

角色：

仓库主控 / 架构守门 / 集成者 / 主执行者。

职责：

1. 读取 CLAUDE.md 和 docs
2. 初始化项目
3. 集成候选代码
4. 按任务卡执行开发
5. 保持项目结构一致
6. 跑自测和类型检查
7. 更新 docs/v0.1/progress.md
8. 对重要决策同步更新 docs/v0.1/decision-log.md
9. 输出交付报告
10. 停在当前任务卡，不擅自进入下一步

CC 可以开发，但必须一次只做一张任务卡。

### 4.4 Codex

角色：

任务卡执行者 / 代码审查员 / 低风险修复者 / 接力开发者。

职责：

1. 读取 AGENTS.md 和 docs
2. 审查 CC 的改动
3. 检查是否越界
4. 检查是否通过验收
5. 检查 progress.md / decision-log.md 是否同步
6. 可接手独立小任务卡
7. 默认采用“审查 + 低风险小修”模式
8. 不重新定义产品方向
9. 可直接修复 typo、路径引用、progress.md、decision-log.md、commit message、测试说明和文档同步问题
10. 低风险小修不需要转回 CC
11. 遇到产品边界、数据模型、状态枚举、新依赖、大重构、删除文件或置信度低于 80% 时，必须暂停并请求用户拍板

Codex 不应和 CC 同时修改同一批文件。

用户拍板不能由 Codex 代签。

## 5. 推荐协作流程

标准流程：

1. GPT / 用户确认产品方向
2. Claude.ai 辅助产出架构或任务拆解
3. CC 在本地仓库执行当前任务卡
4. CC 自测并输出结果
5. CC 更新 docs/v0.1/progress.md
6. 如涉及重要决策，CC 更新 docs/v0.1/decision-log.md
7. Codex 采用“审查 + 低风险小修”模式完成审查闭环
8. 用户确认
9. 提交 commit
10. 进入下一张任务卡

## 6. 提交规范

这是所有 AI 必须遵守的项目级规则。

所有 commit message 必须使用中文描述，commit type 可以保留英文。

英文技术名词可以保留，例如 Vue、Vite、TypeScript、Task、Step、Manual Mode、Prompt、API、BYOK、storage、selftest。

所有 AI 在输出建议 commit message、交付报告或审查报告时，都必须遵守该规则。

规则来源：

```txt
规则：所有 commit message 必须使用中文描述。
来源：docs/v0.1/decision-log.md DEC-011。
影响：所有后续 commit message、AI 建议 commit message、审查输出都必须遵守中文描述规范。
```

---

## 7. 禁止流程

禁止：

1. CC 和 Codex 同时开发同一任务
2. Claude.ai 绕过仓库继续写后续 Step
3. Codex 未读 docs 就直接改代码
4. 任何 AI 一次性实现多个 Step
5. 任何 AI 私自新增 P1 / P2 功能
6. 任何 AI 把 v0.1 扩成泛求职系统
7. 任何 AI 提前接 API / BYOK / 后端
8. 任何 AI 引入复杂状态日志或统计系统
9. 任何 AI 先实现、后补文档
10. 任何 AI 不更新 progress.md 就宣称完成任务

---

## 8. provenance / 规则来源

任何新增规则、约束、口径、红线，都必须写清来源。

推荐格式：

```txt
规则：v0.1 不接 API。
来源：docs/v0.1/decision-log.md DEC-001。
影响：不得新增 OpenAI / Claude / Gemini API 接入。
```

无依据的规则必须写：

```txt
来源：无依据 / 待用户确认
```

不允许凭记忆编造项目规则。

---

## 9. handoff / 接力交接规则

docs/v0.1/progress.md 是项目 handoff 文档。

它用于回答：

1. 当前做到哪一步
2. 哪些任务已完成
3. 哪些任务进行中
4. 哪些任务阻塞
5. 当前是否允许进入下一步
6. 谁执行、谁审查、谁确认

规则：

1. 新会话开工前必须先读 progress.md
2. 每段工作收尾必须更新 progress.md
3. 没有更新 progress.md，视为交付不完整
4. progress.md 与聊天上下文冲突时，以 progress.md 为准
5. 任务状态变化必须同步 progress.md

---

## 10. 决策日志规则

docs/v0.1/decision-log.md 是项目决策日志。

它用于回答：

1. 为什么这么做
2. 谁提出的
3. 谁参与讨论
4. 谁拍板
5. 哪些方案被否决
6. 什么时候可以复审

规则：

1. 重要决策必须记录
2. 重要决策必须带提出者、参与者、拍板者
3. 没有记录的决策，视为没拍板
4. 推翻旧决策必须新增决策记录，不能直接删除旧记录
5. 决策必须说明影响范围和复审条件

---

## 11. 决策与实现同 commit

任何带决策性质的改动，必须在同一个 commit 中更新对应文档。

需要同步更新 docs/v0.1/decision-log.md 的情况：

1. 改产品定位
2. 改 P0 范围
3. 改数据模型核心字段
4. 新增实体
5. 改状态枚举
6. 引入依赖
7. 改存储策略
8. 改 AI 分工
9. 接 API / BYOK / 后端
10. 推翻旧决策

需要同步更新 docs/v0.1/progress.md 的情况：

1. 完成任务卡
2. 任务被阻塞
3. 任务进入审查
4. 用户确认通过
5. 允许进入下一步
6. 发现遗留风险

不允许先实现、后补文档。

---

## 12. 决策权阈值与暂停清单

### 12.1 可以自主处理的事项

1. 修复错别字
2. 修正文档旧路径
3. 补充明显遗漏的引用
4. 小范围调整格式
5. 补充自测说明
6. 修复不影响产品边界的小 bug
7. 根据验收标准补充缺失自测
8. 同步 progress.md
9. 补充已明确需要记录的 decision-log.md 决策
10. 修正不符合中文规则的建议 commit message
11. 修正明显与已拍板 DEC 不一致的文档表述

上述低风险小修由 Codex 直接完成，不需要转回 CC。

### 12.2 必须暂停并询问用户的事项

1. 改产品定位
2. 改 P0 / P1 / P2 边界
3. 改数据模型核心字段
4. 新增实体
5. 引入新依赖
6. 改状态枚举
7. 接 API / BYOK / 后端
8. 大面积重构
9. 删除文件
10. 改任务执行顺序
11. 与 docs/v0.1/decision-log.md 冲突
12. 置信度低于 80%

### 12.3 重大争议处理

重大争议不允许反复碎片化提问，必须一次性输出决策简报：

1. 背景
2. 现状
3. 冲突点
4. 方案 A / B / C
5. 推荐方案
6. 风险
7. 需要用户确认的问题

## 13. 当前阶段安排

当前阶段：

Task 0 已提交完成。当前仅更新项目治理文档，尚未执行 Task 1。

执行者：

CC。

审查者：

Codex。

产品确认：

用户 + GPT。

Task 0 已完成执行、自测、审查、用户确认与 commit。

Task 1 必须由用户另行明确启动，不因本次治理文档更新自动开始。

---

## 14. Step 0 使用决策

Claude.ai 已产出的 Step 0 存储层代码可以使用。

但必须满足：

1. 导入正式仓库
2. 跑通 selftest
3. 跑通 typecheck
4. Codex 审查无越界
5. 用户确认后 commit

如果 Step 0 存储层和 docs 冲突，以 docs 为准。

---

## 15. 冲突处理规则

如果不同 AI 给出的建议冲突，按以下顺序判断：

1. 是否更贴近 v0.1 P0
2. 是否更贴近 Boss 直聘前端求职场景
3. 是否更少引入复杂度
4. 是否能 5-7 天落地
5. 是否更容易被 CC / Codex 接手

优先选择更窄、更稳、更快、更可验收的方案。

---

## 16. 行为基线

必须遵守：

1. 测试失败必须贴失败输出
2. 跳过某步必须说明原因
3. 未验证不得声称完成
4. Deferred / P1 / P2 不许顺手做
5. 发现文档冲突必须先报告
6. 能本地判断的不要来回传话
7. 低置信度不要硬拍
8. 不允许“顺手优化”扩大范围
9. 不允许用猜测替代文档事实
10. 不允许把失败包装成成功

---

## 17. 项目纪律口诀

文档定方向，任务卡定范围，验收线定完成。  
AI 只能施工，不能改地皮。  
先让工具能自用，再让项目变漂亮。  
没有记录的进度，视为没完成。  
没有记录的决策，视为没拍板。
